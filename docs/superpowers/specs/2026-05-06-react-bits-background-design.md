# React Bits 背景动效替换设计

## 目标

将博客当前全站共用的宇宙星点背景替换为更克制的科技感动态背景，并保持文章阅读体验优先。

本次改造需要满足以下要求：

- 全站统一使用新的背景动效，包括首页、文章列表、文章详情页和关于页。
- 新背景需要具备明确但克制的科技感，不能喧宾夺主。
- 背景必须继续作为内容层之下的固定铺底，不影响现有页面结构和点击交互。
- 站点仍然需要以 Astro 静态站点形式构建并部署到 GitHub Pages。

## 非目标

- 不在本次改造中调整正文布局、导航结构或卡片样式。
- 不引入强交互型背景，例如鼠标驱动的大幅波动、吸附粒子或高频镜头运动。
- 不为不同页面做差异化背景版本，全站保持统一实现。

## 方案比较

### 方案 A：React Bits `Threads`

这是本次推荐方案。

优点：

- 线性流动纹理有明确的科技感，但整体密度较低，适合作为阅读型页面背景。
- 背景表现更偏“电流/纤维/数据流”，与现有深色宇宙主题兼容。
- 透明铺底更容易与现有 `body` 渐变和网格叠层融合。

代价：

- 需要引入 React 集成和额外运行时依赖。
- 需要对默认参数进行收敛，避免动效过亮或过快。

### 方案 B：React Bits `FloatingLines`

优点：

- 视觉更强，展示感更明显。
- 与项目现有 `three` 依赖的方向一致。

代价：

- 对文章详情页来说存在感偏强，更容易分散阅读注意力。
- 在全站统一使用时，风格会更接近展示页而不是内容站。

### 方案 C：React Bits `Grainient`

优点：

- 阅读干扰最小，性能风险也较低。
- 更容易做出统一且稳定的氛围。

代价：

- 动态科技感偏弱，不足以替代当前背景作为站点主视觉。

## 最终决策

采用方案 A：基于 React Bits 的 `Threads` 背景作为全站统一背景。

实现上不直接照搬演示参数，而是将其调整为“低速、低对比、冷色系”的阅读型版本：

- 主色调使用青蓝和偏白高光，避免高饱和紫色或暖色。
- 降低线条密度、速度和亮度，优先保证正文可读性。
- 不启用鼠标驱动交互，减少注意力争夺与移动端行为差异。
- 保留当前页面底层渐变氛围，但削弱扫描线叠层强度，让新背景成为主动态层。

## 架构设计

### 总体结构

背景实现将从“Astro 模板 + 独立浏览器脚本”的模式切换为“Astro 外壳 + React island 背景组件”的模式。

计划中的职责划分如下：

- `BaseLayout.astro`
  继续作为全站外壳，负责挂载背景组件、页头、主体内容和页脚。
- `CosmicBackground.astro`
  保留为 Astro 层入口，但内部从 `canvas + 外链脚本` 切换为 React 背景组件容器。
- React 背景组件
  新增专用 TSX 组件，封装 `Threads` 动效及其参数，负责在浏览器侧初始化和渲染。
- 全局样式
  继续控制背景层级、固定定位、内容层级与扫描线叠层样式。

### 渲染方式

背景组件将作为客户端组件挂载。页面首屏仍然由 Astro 输出静态 HTML，背景在浏览器端加载后开始动画。

这符合 GitHub Pages 的静态部署模型，因为最终部署产物仍然是静态资源，只是在客户端执行背景脚本。

## 组件与文件边界

计划中的主要变更边界如下：

- 修改 `astro.config.mjs`
  接入 Astro React 集成。
- 修改 `package.json`
  增加 `react`、`react-dom`、`@astrojs/react` 以及 `Threads` 所需依赖。
- 修改 `src/components/CosmicBackground.astro`
  移除旧的 `canvas` 结构，改为渲染 React 背景组件与扫描线层。
- 新增 React 背景组件文件
  负责承载 `Threads` 逻辑与项目级参数。
- 修改 `src/layouts/BaseLayout.astro`
  去掉旧的 `/scripts/cosmos.js` 注入。
- 修改 `src/styles/global.css`
  调整背景层、扫描线叠层和内容层级，确保新背景不盖住正文。
- 删除 `public/scripts/cosmos.js`
  旧背景脚本不再使用。

## 视觉与交互约束

为保持“克制科技感”，实现必须遵守以下约束：

- 背景动画只作为铺底氛围，不承载功能性信息。
- 运动方向和频率要平稳，不能出现明显的闪烁、爆发或强节奏脉冲。
- 页面滚动、鼠标移动和触摸事件都不应该成为背景主要驱动源。
- 扫描线与网格叠层保留，但透明度需要下调，避免与 `Threads` 线条叠加后产生噪声感。
- 移动端需要继续保持完整背景，但在必要时可降低视觉强度或计算量。

## 性能与降级策略

### 性能目标

- 首屏内容必须在背景脚本加载前即可正常阅读。
- 背景层不能阻塞正文渲染或交互。
- 动画参数应优先面向中低端移动设备收敛，而不是追求桌面端展示效果。

### 降级策略

- 如果背景组件在浏览器端初始化失败，页面仍应保留现有 `body` 渐变和静态叠层，不影响内容访问。
- 如果某些设备上背景性能不佳，可以通过参数降低密度或分辨率，而不是回退到更复杂的交互效果。

## 测试与验证

需要完成以下验证：

- `npm run build`
  确认 Astro 静态构建通过，React 集成与背景组件可以被正确打包。
- `npm test`
  确认现有测试不因布局改造而回归失败。
- 本地手工验证
  检查首页、文章列表、文章详情页和关于页的背景是否统一生效。
- 响应式验证
  检查桌面端和移动端视口下正文可读性、滚动流畅度与页头层级。
- 视觉验证
  确认新背景不会压过正文卡片、导航和页脚文字，扫描线强度合适。

## 风险与处理

### 风险 1：React Bits 示例参数过于演示化

处理方式：将背景封装在项目自有组件中，由项目自行控制参数，不直接使用演示站默认视觉。

### 风险 2：客户端组件引入后首屏层级错乱

处理方式：明确背景层、扫描线层、页面壳层三者的 `z-index` 分工，并通过全站页面手工验证。

### 风险 3：GitHub Pages 部署后静态资源路径异常

处理方式：保持 Astro 现有 `site`、`base` 和 `output: "static"` 配置模式不变，仅在框架层增加 React 集成。

## 实施结果定义

当以下条件全部满足时，本次改造视为完成：

- 全站旧宇宙星点背景被移除。
- 全站统一显示低干扰的 `Threads` 风格背景。
- 正文阅读体验不因背景替换而明显下降。
- 项目能够继续通过本地构建，并保持 GitHub Pages 静态部署兼容。
